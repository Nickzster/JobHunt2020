Pass By Value

By default parameters in C++ are passed to functions by value. What this means is that when you pass a parameter, you are actually passing a copy of the value. Lets look at an example:

 


 

In the above example the variable x is declared and initialized with a value of 3. The program calls the function foo and takes with it, a copy of the value that is contained within the variable x (3 in this case). When the flow of the program reaches the function foo, a new local variable is declared. It is called z and has a type of int. This local variable is somewhat special in that it gets initialized with the copy of the value that was passed to it (3) in this case. 

In the section on functions you saw that the stack was used to pass parameters. While that animation is somewhat accurate it doesn't tell all of the details. Here, a little more detail will be given. Of course this is not an assembly language class but sometimes seeing the details will give you a better understanding of things. The stack if you do not know it is a piece of memory that gets created when your program runs. It is used to hold local variables and information that  allow a function to return back to where it came from. The stack has basically two operations, push and pop. The push operation is used to put things on the top of the stack. The pop operation is used to remove things from the top of the stack. The stack then is called a LIFO (Last In First Out) structure in that data is always added and removed from the top of the stack. This type of data structure is quite efficient because it can contain no gaps, meaning that the next piece of information is always in the next memory location. 

Some of this might be beyond the scope of this class but I believe it is important to get an idea of how the stack is used so that you can truly understand how parameters are passed. If you take the assembly language class you will learn that the CPU consists of several registers. Some of these registers are used for specific things and others are used for general purpose.  

The first register we need to get a feel for is called the ESP (Extended Stack Pointer) and it holds the address of the top of the stack. This address is used so that data can be pushed and popped at the right location. The EBP register (Extended Base Pointer) is used to hold a reference to the bottom of the stack. 

One other register needs to be mentioned here and that is the EIP (Extended Instruction Pointer) which holds the address of the next instruction to be executed. 

When a function is called a few things need to happen:

  • The function parameters are pushed on to the stack. In the above example the value in the variable x is pushed onto the stack (3 in this case)
  • The functions return address is pushed onto the stack (This is the address of where the function needs to come back to) This is also the contents of the EIP.
  • The frame pointer or previous value of the EBP  is pushed onto the stack
  • The address of the function is placed into the EIP (Instruction Pointer) and the code jumps to the function.

In order for the function to return back to where it came from some of the information placed on the stack needs to be restored. The first thing that needs to be copied back is the EBP. In order to do this though the variable z needs to be popped off the stack. In other words this variable gets destroyed. Once the EBP is copied back the return address needs to be popped into the EIP so that the code can branch back to where it came from. In the animation what is left at the top of the stack is the variable x. This makes sense because the context of the program is back in the function main and it needs to be able to have access to the variable x. 

In this class you really don't need to know how the stack works to get by. What is important is that you understand that when a variable is passed by value a copy of it gets sent to the function where it is placed in the formal parameter. 

You should also note that since the formal parameter is created on the stack it must also be removed from the stack in order for the function to return back to where it came from. Remember, the return address and the EBP were all placed on the stack before the variable was created. What this means is that local variables are destroyed when the function returns. In other words they no longer exist.

Pass By Reference

There are times when you may want a function to produce more than one value. This may seem impossible or at least problematic because, as  you may have noticed functions can only return one value. The way to get around this limitation is to pass the values by reference.  When you pass parameters by reference what is passed is the address of where the variable sits in memory.  Parameters are still passed to the function by value but when passing by reference, a copy of the address of where the variable is located in memory gets passed. Passing values by reference is really an easy thing to do.  All you have to do is put an & in front of the formal parameter that you wish to pass by reference:

 

 

The parameters are passed the same way as they would be with pass by value. When the parameter in the function gets declared, it now refers to the memory location that was passed to it.

Code Review

As you can see, The values of the variables actually do get swapped. Again, this is because using pass by reference the variables x and y of the formal parameters in the function swap actually refer to the same memory locations as the variables a and b that were declared in main. It may be easier to think of x and y as aliases or nicknames for the variables a and b. It is like being able to refer to the same value with two different names. In the function swap, you would use the nickname and in main, you would use the given name.

One other thing to note is that like passing parameters by value the stack must be popped so that the return address can be popped into the EIP.

Downside

Many people feel that passing by reference can be a bit on the confusing side. Suppose that you are a member of a team of programmers working on a project and you have to use a function someone else wrote. If they don't give you the exact details, you would not know by looking at a call to the function that the parameters are actually being passed by reference. So, you would have no idea that a function may actually modify your copy of the variables. In my opinion this is somewhat of a weak argument but it exists none the less. Of course, this can always be remedied through the use of pointers which we will go into in great detail in a future lesson.

Overloading Functions

Before object oriented programming hit the scene, if you had a function that you wanted to one time work with doubles and the next time work with chars, or maybe one time you needed to pass 2 ints and a double one time but the next time you only needed to pass two ints to function then you were forced to create a different function for each version that you needed. Through overloaded functions, you can almost achieve this. You still have to write the function bodies but all of the functions have the same name. Lets look at an example:

Code Review

Notice that there are two addEm functions defined and that they are only different by their function header. In main the first call to addEm has only two actual arguments while the second call has three actual arguments. The compiler will sort out which function to call by matching arguments.

NOTE:

The compiler will always try to match arguments in the best way it can. For example, lets say that you have a function that takes two doubles but you pass it two ints; the compiler will convert the ints to doubles automatically. This type of conversion is called automatic type conversion.


Code Review

 

In the above code example, the call to the swap function takes two doubles but main is passing it two int.The compiler will automatically convert the ints to doubles. You should always take care to provide function bodies for all argument types that you intend to use.

Automatic Type Conversion Rules:

 

The compiler will try to resolve overloaded parameters in the following order

  1. Exact match
  2. Matches using promotion within integer types (short to int, int to long, etc), or floating point types (float to double). Note: char to int and bool to int are considered promotions also
  3. Matches using other conversions of predefined types, such as int to double
  4. Matches using user defined types (Discussed later)

Note:

Be careful for ambiguous overloading

The declarations of the functions (although confusing) are perfectly legal. It is the call to the function from main that will force an error because the compiler cannot figure out which version of the function should be invoked.
