Structures

 

Arrays gave us an introduction to abstract data types. In this lesson, we are going to take that one step further. A structure is an abstract data type that you the programmer define. Atomic types like int, char, double, etc are defined for you. You can use any of these types and combine them to create a structure. Unlike arrays, members of structures are not homogenous. This means that you can mix as many types as you need to create a structure. You can also have an array of structures that you define. Lets look at the syntax:





A structure is defined with the keyword struct followed by a name that you the programmer give it. The body of the structure should be surrounded by braces with a semicolon after the last brace. The missing semicolon is a common mistake; be careful. You give your structure a meaningful name just like you would with variables. The variables within the body of the structure are called member variables and you can put any valid C++ type within the open and closed brace of the structure. You can even put a structure there.

Just a definition

Beginning programmers have sometime have trouble understanding that a structure like the one above is nothing more than a definition. This means that it occupies no space in memory. It is used by the compiler to calculate the amount of space needed for the structure.

Header Files

In the days before C++, 'C' programmers were very familiar with the use of header files. Since C++ Level I you have undoubtedly gotten used to doing something like this at the beginning of your files:

#include <iostream>

In C and the early days of C++ you would include a header file like this:

#include <stdio.h>

Even though things have changed, we still use header files in C++. You can do the same thing the old C programmers used to do but it is not considered appropriate by today's standards. 

Starting now every definition you create should go in a header file. You will then include that header file into your cpp file. As you have probably deduced from above, header files have the extension .h 

The reason that you want to use header files is so you can include it in any file that is part of your project. Remember, C++ requires a definition of functions, structures, classes, etc. before you can use them. 

Lets' look at an example:
Definition in a header file
This header file contains the definition for a struct called PERSON. Now any .cpp file that needs the definition only needs to include the header at the top of the file
Header included in main
Notice that the main file now uses a pre-processor directive to include person.h

Notice that main.cpp uses the preprocessor directive #include to include the header file. When the compiler opens main.cpp in will go through person.h and make note of the definitions that are defined within it. It is like taking the contents of person.h and pre-pending it to the beginning of main.cpp. Notice also the include statement in main.cpp uses double quotes instead of the brackets <> . Header files that are part of the C++ compiler path use the brackets:

#include <stdlib.h>
#include <conio.h>

You should note that you can add your header to the compiler path for it's headers and use the brackets but this is not something that is done too often.

Header files that you create and add as part of your project get double quotes

#include "myHeader.h"

You can also include the path within the double quotes

#include "c:\\myheaders\\myHeader.h"

As a word of caution, if you are going to include the path then you should avoid absolute paths and use a relative path instead:

#include "..\\myheaders\\myheader.h"

Including Definitions Once

 When programs start getting big it may become difficult to figure out if a header file has been added to a .cpp file or not. Now you may be wondering how this could happen but in some cases someone will create a file called includes.h which contains all of the headers being used in a project.  

Something like this:


Then in a .cpp file they simply 

#include "includes.h"

Anytime a new header needs to be used it is simply added to the includes.h file. Ok, this doesn't seem to be a problem at this point but what happens if someone new comes along and does not realize that includes.h is there and they do this:

#include "includes.h"
#include "Person.h'

Now the persons.h file has been included in the .cpp file twice. So the definitions are duplicated. We really don't want to have things like this happen so we need to use a mechanism to guard against it. 

The mechanism we are going to use is borrowed from the preprocessor and it works like this:

#ifndef somelabel
#define somelabel

#endif

somelabel can be just about anything that you want it to be. The typical thing is to give it the name of the file in CAPS. 

So how does this work? When the file is opened the preprocessor checks to see if somelabel is not defined. It it is not, it is then defined by the preprocessor. If later on the the file is opened a second time from the same .cpp file everything within the ifndef will not execute because some label was already defined by the previous opening of the file. Data and Functions

Switching to object oriented programming can be a difficult transition for some people. The thing to keep in mind is now classes are data centric. This means that a class will have a data section and a set of functions that operate on that data. Before going into this in great detail it may be best to first look at how a class is defined.

 

 

 

Classes look a lot like structures. That is why we talk about structures first. As a matter of fact C++ really means C with classes. When Bjorn Stroustrup invented the C++ language, he really just made modifications to the C programming language. If you look closely you will notice that a class is nothing more than a struct that allows you to put functions within it. Of course that is a very simplistic statement but pretty accurate.

 Encapsulation

 

 One of the big things with object oriented programming is a term called encapsulation. This really means that we want to hide how the functionality is implemented. There are several reasons why this is a good idea. One is that it allows programmers that are using your code to use it without having to worry about the details of it (You don't need to know the details of how your car works to drive it do you?). Another reason is that maybe you don't want people to see how your code is written or the details involved.  One way C++ achieves encapsulation is to have a private section of the class. By default everything you put in a class is private unless you specifically state that you want it to be public. C++ like most programming languages will always default on the side of good practice.

 This brings us back to the idea of object oriented programming being data centric. In most cases the data of the class will be located in the private section. This way it is hidden from the outside world. You can then create public functions that operate on the data. In this way you control how the data is accessed and modified.

Public and Private

You specify which sections of the class that you want to be public and private by using the keywords public, and private

 

 

 

You might say the private section is the data section. Data that you want to be private gets defined after the colon of the private statement.  Things that you want to be public go after the colon of the public statement. The order of the statements doesn't matter, you can put public before private if you want.

 So what is meant by private? Well, it means that nothing outside of the class can access it. You normally put things that make up the classes data in the private section. These things typically consist of variables but you will also want to place utility functions in this section also. Utility functions are functions called by other functions from within the class. Basically things that the outside world does not need access to but are needed for your object to perform it's duty.

Prototypes

In C++ Level 1 you learned to prototype all of your functions before you use them. Nothing has changed at this level. You still need to prototype all of your functions. Now though, you need to put them as part of the class. The rectangle class then has prototypes for two functions: setValues and area. Both of these functions operate on the data section. The data in this case is the length and the width of the rectangle.

 

Now that the class has been defined the body of functions need to be written. Again, this is very similar to what you did in C++ level 1. You prototype your functions, then you create the body of them. The same thing has to happen here with a slight difference. Now, you have to tell the compiler what class the functions belong to. Basically, where the prototypes for the functions reside.

 Scope Resolution Operator

On the right in the above example is the code for the Rectangle class. It sits in it's own .cpp file. The definitions for the class are brought in by including the header file. Notice the function header for the function area:

int Rectangle::area()

 To tell the compiler what class the function area belongs to, the scope resolution operator is employed. This is nothing more than the double colon:

::

 It tells the compiler the function area is a member of the Rectangle class. Without it:

 int area()

The compiler would see this as just another function and your code would not compile because you have not created the functions that are defined in the Rectangle class.

 Remember from the introduction that classes are blue prints for objects. This means that a class does not really occupy any memory. A class is also a definition for a type and gets created just like any other variable. To turn a class into an object you must create an instance of it. This is called instantiation and is very easy to do. Just like creating variables of type int:

int x,y,z;

Rectangles can be created in the exact same way:

Here three instances of Rectangle have been created. They are called x,y,z. So, it can be said that their identity is x,y,z respectively.

Accessors and Mutators

The private section of a class should have functions that are used to access it. In this way the programmer controls how the data gets set. An example might be of a time class that keeps track of the current hour minute and second. If someone has direct access to the data they could set the hours minutes and second to whatever they want. It doesn't seem appropriate to allow someone to set the the seconds above 60 because there are only 60 seconds in a minute. It would seem logical that if the seconds were set to 61 then 1 would be added to minutes and the seconds would be set to 1 also. This would be difficult to control without a function to handle.

One of the things your want to do for good object oriented programming practices is to provide public functions to private data. These types of functions are called accessor and mutator functions.

  • Accessor - is a function that accesses or returns data from the private section.
  • Mutator - is a function that modifies the data in the private section.

More On Accessors

In object oriented programming methods are functions that are part of the class. In C++ the term function has kind of stuck. You may see these two terms used interchangeably. While some C++ purists might object to calling a function a method I personally don't see the harm in it.

Constructor Basics

It has been said that a class is really a blueprint for an object. It also goes that the constructor is the factory that builds the object. We have seen that classes typically have a data section that is defined as private. We have also seen the value of having a public interface to the private data but what is the value of the variables that make this private section when the object is created?

Let's examine the Rectangle class again as an example:

In main if we were to create and instance of the rectangle:

 Rectangle rect;

 cout << rect.area() << endl;

 What would print? The answer is that it would be some kind of garbage. This is because in C++ when variables are created, they contain whatever happens to be in the memory location at the time. In most instances this is nothing more than garbage.

One solution to fixing the problem is to call the setValues function:

rect.setValues(5,7);

Which works but it required two function calls to simply setup the variables. Also, there is no way of guaranteeing that if someone else uses the class you wrote that they are going to remember to call the setValues function before trying to use the object instance. This is where the constructor comes in.

The constructor is a special function that gets called one time when the object gets created. This function gets called automatically. You don't have to do anything. If it is called automatically then you may be wondering what happens if you don't supply a constructor. The answer is the compiler tacks one on to your code. Now you cannot see this code or take advantage of it but rest assured if you don't supply a constructor one will be created for you. 

Three Things You Need To Know

  1. *Constructors are always defined as public
  2. Constructors have the same name as the class
  3. Constructors have no return type not even void. 

* It is possible to have a private constructor but when you do this there is no way for you to construct the object publicly.  This will be covered in more detail in the sections on composition and inheritance.

Constructor Notes

Calling Constructors:

Using the default constructor in C++ has always seemed a little odd to me. It seems you should create the object like this:

Rectangle rec();  // This is not correct!!!!

This seems correct because you call the overloaded constructor like this:

Rectangle rect(2,3);

Look closely at the default constructor and try to determine what it looks like. The compiler will see this as a funciton prototype of a function called rec that has no arguments and returns a Rectangle object. This the reason the default constructor is called without the use of the paraentheses:

Rectangle rect;

More Calling Constructors

On the surface it seems that you should be able to recall a constructor like like any other function:

Rectangle rect;
rect.Rectangle(5,6);

But you are not allowed to do this. Remember the constructor gets called automatically when an instance of the object gets created.

Reinitialization

One thing you can do is make an explicit constructor call:

Rectangle r(3,5);
r = Rectangle(4,5);  // Create ananomous object

What the above code does is create what is called an ananomous object. It is called this because it is returning an object that has no name or reference. What happens is that this object gets assigned back on top of the previously created Rectangle object.

I should point out that this is something that is not to common to do. In the case of the Rectangle object it would be just as simple to call the setValues function. This would accomplish the same tasks with much less overhead.

Constructor Finally

Before the discussion on Constructors is concluded it is important to point out one thing. It was said that if you do not supply a default constructor the compiler will create one for you. This is true except in the case of overloaded constructors. If you have overloaded constructors you are also going to have to create the default constructor. If you fail to do so you are going to get a compiler error.
