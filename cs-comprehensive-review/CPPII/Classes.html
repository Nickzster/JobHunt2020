Data and Functions

Object Oriented programming (OOP) is a different approach to solving problems than we have seen so far. In the level 1 class we used functional decomposition, where functions were at the center of program organization. The problem with this approach is that there is no way of bundling data with operations. This is going to be our focus for programming from now on.

If you read the introduction then you know that an object is the vehicle used to bundle data together with functionality. The question is, what is an object? Put simply, an object is an instance of a class. To refresh your memory you can create an instance of a primitive data type simply by creating a variable:

int myInt;
double myDouble;
char myChar;

In the above three instances were created. An int named myInt, a double named myDouble, and a char named myChar.

An object is an instance of a class. A class is a blueprint for an object. This means the compiler uses the class to build what you have defined in memory. In essence you are creating your own data type. We call this a complex type because it is not built into the language the way primitive data types are.

Switching to object oriented programming can be a difficult transition for some people because it is truly a different way of thinking. It was stated earlier, and it is important to keep in mind that classes are data-centric. This means that a class will have a data section and a set of functions that operate on that data.

You may recall that objects have three characteristics:

  1. State
  2. Behavior
  3. Identity

Before going into great detail on the characteristics of an object it may be best to first look at how a class is defined syntactically.

In the level 1 course we studied structures and how they were good for bundling like information. As an example consider the need to store information about a single person. An efficient way to group all of this information would be to place it inside a struct:

struct Person
{
string firstName;
string lastName;
string email;
};

If you look closely you will see that classes look a lot like structs. As a matter of fact C++ really means C with classes. When Bjorn Stroustrup invented the C++ language, he really just made modifications to the C programming language. Notice that a class looks like nothing more than a struct that allows you to put function prototypes within it. Of course that is a very simplistic statement but pretty accurate. If you need a review of structures, you may want to review the C++ level I lecture notes which can be found here.

Encapsulation

One of the big things with object oriented programming is a term called encapsulation. The word encapsulation really means bringing together. In object oriented programming we are bundling data and functions that operate on that data together into a class. One of the good things about encapsulation is that it allows you to hide the details of how a class is implemented. This allows programmers that are using your code to use it without having to worry about the details of it (You don't need to know the details of how your car works to drive it do you?). By default everything you put in a class is private unless you specifically state that you want it to be public. C++ like most programming languages will always default on the side of good practice.

This brings us back to the idea of object oriented programming being data-centric. A class is comprised of a data section and functions that operate on that data section. As a general rule, if you create public functions that do not operate on your class' data section then you are probably doing it wrong. The data section of a class are considered to be the attributes of its class. In most cases the data (attributes) of a class will be located in the private section. This way it is hidden from the outside world. You can then create public functions that operate on the data. In other words, a public interface to private data. In this way you control how the data is accessed and modified.

See what LearnCPP has to say about access and encapsulation

Public & Private

Class definitions can be divided into three separate sections. They are

  1. public
  2. private
  3. protected

These sections are divide by using one of the keywords followed by a colon. Note that it does not matter which order you place the statements. The example here shows private then public but it could just as very well be public followed by private:

The private section is where the data or attributes of the class should be located. Anything that you do not want outside classes or functions to have direct access to should be placed in the private section of the class

The attributes (data) of the class should be placed in private because you want to be able to control how they are accessed and modified. You will normally create a function in the public section to access the private data section. Again, this is to create a public interface to private data.

An Example

As an example let's take a look at a class called Rectangle. The first thing we should do is define the attributes (data) of the class. We could get into greater detail with this but simply put a Rectangle consists of a length and a width. This will make up our data section. The last thing we need to consider is the functionality we want our Rectangle to have. We need a function that will set the attributes of the class and another that will compute and return the area. We can say then say that these two functions define the behavior of the class.

// Rectangle.h
#ifndef Rectangle
#define Rectangle

class Rectangle
{
private:
   int length, width;

public:
   void setValues(int len, int wid);
   int area();
};

What you see above is the class definition, or blueprint, for a new complex type called Rectangle. It is a complex type because it is a class type that is defined by the programmer. Notice that the private section contains the data (attributes) of the class. Because they are private they can only be changed by members of the same class. In other words only the setValue and area functions can change the length and the width but nothing else

You couldn't directly modify these variables from another function. The following example depicts this:

int main()
{
    Rectangle rect;  // Create an instance of a Rectangle called rect
    cout << rect.length << endl;     // This is an error because length is private and main is not a member of Rectangle
}

Classes are typically put inside a header file because the compiler needs to see a definition of everything before it compiles it. If you want to use cin or cout for instance, you need to include iostream. The same holds true for the classes you create.

The compiler treats all files individually, meaning it doesn't know anything about what is going on inside other files of your project. Since all files are treated separately you need to have a mechanism to show it the definitions in each of the files where you intend to use the functionality. This where the header file comes in. You can simply put all of your definitions in a header file then include the header file in any .cpp file where you intend to use the functionality. This is much better than copying and pasting these definitions inside every file where it's not located.

Creating and using header files will be covered in the next section.

Note on the Use of the Word Definition

OK, some people are probably going to be really infuriated by the fact that I use the word definition throughout this material. Technically speaking a function definition is the actual body of the function. This should be distinguished from its prototype which I have been calling a definition. I understand the difference but what I am getting at is that much of this stuff is a way to define what it is to the compiler... in other words, a definition. A function prototype defines what the function is, including its inputs and outputs to the compiler. Please excuse me if you feel I am using the word in error. It just seems to be the word that best fits to me.


Just Prototypes!

In C++ level 1 you learned to prototype all of your functions before you use them. Nothing has changed at this level. You still need to prototype all of your functions. Now though, you need to put them as part of the class. The Rectangle class then has prototypes for two functions: setValues and area. This means that the body or function defintions need to be written. This is normally done in a .cpp file

Creating function bodies is very similar to what you did in C++ level 1. You prototype your functions, then you create their body. The same thing has to happen here with a slight difference. Now, you have to tell the compiler what class the functions belong to. Basically, where the prototypes for the functions reside: 

// Rectangle.cpp
#include <iostream>
#include "Rectangle.h"

void Rectangle::setValues(int len, int wid)
{
   length = len;
   width = wid;
}

int Rectangle::area()
{
   return length * width;
}

The Scope Resolution Operator ::

The example above contains the code for the functionality of the Rectangle class. It sits in its own .cpp file. The definition for the class is brought in by including the header file. Notice the function header for the area function:

int Rectangle::area()

The scope resolution operator is employed to tell the compiler what class the function area belongs to. The scope resolution operator is nothing more than the double colon: :: It tells the compiler the function area is a member of the Rectangle class. In other words it tells the compiler that the prototype for the area function can be found within the Rectangle class. Without it:

int area();

The compiler would see this as just another function and your code would not build because you have not created the functions that are defined within the Rectangle class

When you first start creating classes and writing functions that belong to these classes you may find it easy to forget to use the scope resolution operator to define the function to the class. Keep this in mind as you move along.


Classes to Objects

Remember, classes are blue prints for objects. This means that a class does not really occupy any space memory. A class is also a definition for a type (a complex type) and gets created just like any other variable. To turn a class into an object you must create an instance of it. This is called instantiation and is very easy to do. Just like creating variables of type int:

int x,y,z;
Rectangles can be created in the exact same way: 
Rectangle x,y,z;

Here three instances of Rectangle have been created. They are called x,y,z. So, it can be said that their identity is x,y,z respectively.

Here is an example main function that uses the Rectangle class:

#include 
#include "Rectangle.h"

using namespace std;
int main()
{
Rectangle r1, r2;  // Two instances of the Rectangle are created

    cout << "Enter a length and a width for the first rectangle" << endl;
    int l, w;
    cin >> l >> w;
    r1.setValues(l, w);
    cout << "The area of the first Rectangle is " << r1.area() << endl;

    cout << "Enter a length and a width for the second rectangle" << endl;
    cin >> l >> w;
    r2.setValues(l, w);
    cout << "The area of the first Rectangle is " << r2.area() << endl;

    return 0;
}

The output from the above code would be:

Enter a length and a width for the first Rectangle
5 6
The area of the first Rectangle is 30
Enter a length and width for the second Rectangle
7 8
The area for the second Rectangle is 56
Press any key to continue . . .

Messages

In object oriented programming it is said that you send an object messages. A message is nothing more than a request for an action. The message request is sent to the object by separating the two with the . (dot) operator. You can provide additional information to the object by placing it inside the parentheses:

The reality here is that this is nothing more than invoking (calling) a function that is a member of an object. You simply need to specify which object you want the function call to go to.

In the case of the Rectangle the setValues function can be invoked in this way:

Rectangle rect;
rect.setValues(5, 6);  // Send the rect object the setValues message.

Some of this type of terminology comes from newer OOP languages like Java. It is important to note that if you are using pointers then you still need to use the arrow operator. 

void printArea(Rectangle *rect)
{
    cout << rect->area() << endl;
}

Accessor and Mutators

A class should have public functions to access the data within its private section. In this way the programmer controls how the data gets set. An example might be of a time class that keeps track of the current hour, minutes, and seconds. If someone has direct access to the data they could set the hours minutes and second to whatever they want. It doesn't seem appropriate to allow someone to set the seconds above 60 because there are only 60 seconds in a minute. It would seem logical that if the seconds were set to 61 then 1 would be added to minutes and the seconds would be set to 1 also. This would be difficult to control without a public function to handle this logic. A public function would allow the user to change the private data section while preventing changes that may have adverse effects.

It is good object oriented programming practice to provide public functions to private data. These types of functions are called accessor and mutator functions.

  • Accessor - is a function that accesses or returns data from the private section.
  • Mutator - is a function that modifies the data in the private section.

Constant Accessors

An accessor is meant to simply return private data in one form or another. This means that it should not modify private members of the class in any way. Because of this it is appropriate to make accessors constant. You may be saying to yourself that the area function is an accessor and it has no const declaration. If you were thinking this you would be right. The area function can be improved by using the const keyword:

int Rectangle::area() const
{
   return length * width;
}

Don't forget that this declaration must also go with the prototype inside the header file:

class Rectangle
{
private:
    int length, width;

public:
   int area() const;
   void setValues(int len, int wid);
};

The above const declaration ensures the function will not modify the data section in any way. If this is not completely clear hang on. We are going to be discussing this when we get to passing parameters.

Functions & Methods

In object oriented programming methods are functions that are part of the class. In C++ the term function has kind of stuck. You may see these two terms used interchangeably. While some C++ purists might object to calling a function a method I personally don't see the harm in it.

Creating a class

A Brief Recap

Classes are blueprints for objects. They are the definition to the compiler as to what gets built in memory when you create an instance of it. An instance of a class is an object.

Class definitions are typically placed in header files and the corresponding code is placed in a .cpp file. You should never put source code inside a header file because it will be duplicated in each file it is added to. Definitions go inside header files.

Definitions are things like classes, function prototypes, and const variables.

A class has a private section which contains the attributes (data) of the class. It also has a public section that contains functions that are used to access and modify the attributes that make up the private section.

The functions that are part of the public section are nothing more than the function prototypes for functions that need to be created else where.

The functions that are defined in the class need to be created in a .cpp file. These functions are very similar to any other function except that you use the scope resolution operator to specify that the function belongs to the specific class.

There are two types of functions - mutators which modify private data, and accessors which return private data.

When you create instances of classes (objects) you use the dot operator to invoke the functions of the class.


What is this?

Before getting into the meat of constructors I want to pause for a second and take a look at the potatoes: accessing members of the current object from within the object. Most OOP languages have a mechanism for accessing other members. In C++ we use the this pointer.

This is a special type of pointer that refers to members of the class it is contained within. You might say that this refers to itself. Here is an example:

#include <iostream>
#include "Rectangle.h"

void Rectangle::setValues(int length, int width)
{
    this->length = length;
    this->width = width;
}

int Rectangle::area()
{
    return this->length * this->width;
}

If you look carefully you will notice that the formal parameters to the setValues function have the same name as the data of the class (length and width). Through the use of the this pointer you can distinguish between the variables that are part of the class and the formal parameters. Without using the this pointer your code would look something like this:

#include <iostream>
#include "Rectangle.h"

void Rectangle::setValues(int length, int width)
{
    length = length;
    width = width;
}

If you remember the level 1 discussion on scope you know that the example code simply assigns the formal parameters to themselves. This is not very good for setting the attributes of the class.

Many have the philosophy that any time you access a function or a variable of the class from within the class you should use this as a prefix. It helps distinguish what is a member of the class from what is not. There are many other uses for the this pointer. We will see more of it as we move ahead.


Constructor Basics

It has been said that a class is really a blueprint for an object because when you create an instance of that class you get an object in memory that has all of the components of that class. The class constructor then is the factory that builds the object because it initializes all of the attributes of the object. We have seen that classes typically have a data section that is defined as private. One problem with what we have seen so far is that when the object is created the value of the variables is garbage. In the case of the Rectangle if we call the area function before calling setValues we are going to get a pretty ugly output. The point here is that we need a mechanism to initialize the variables to some reasonable value when the object gets created. This is the job of the constructor.

You may think that it is really no big deal to call the setValues function before using the class and in this case I might agree with you. You have to think big picture and the idea of creating more complex classes that may be used by others. A person that uses your class may simply start using some of the accessor functions without calling a function to initialize first. It is always a good idea to create a constructor for your code to guard against things like this.

If you do not create a constructor the compiler will tack one on to your compiled code. You can't see it or use it but it is there nonetheless.

See what LearnCPP has to say about constructors

Three Properties

Adding a constructor to your class is an easy thing do as long as you can remember the three properties that make up a constructor:

  1. *Constructors are always defined as public
  2. Constructors have the same name as the class
  3. Constructors have no return type not even void

 

*It is possible to have a private constructor but when you do this there is no way for you to construct the object publicly. This will be covered in more detail in the sections on composition and inheritance.

At this point let's take a look at adding a constructor to the Rectangle class:

// Rectangle.h
#ifndef RECTANGLE
#define RECTANGLE

class Rectangle
{
    private:
        int length;
        int width;
    public:
        void setValues(int length, int width);
        int area(); Rectangle();// Constructor definition
};

#ifndef

After the prototype for the constructor is added to the class definition it then needs to be add to the .cpp file. This isn't any different than adding any other function to the class. You simply need to make sure you adhere to the three properties of a constructor: 

// Rectangle.cpp

Rectangle::Rectangle()
{
    this->length = 0;
    this->width = 0;
}

Now if the area function is called before setValues the output will be 0 and not garbage.


Initialization List

In the above example the attributes length and width get set to 0 in the body of the constructor. There is an alternate and preferred way to initialize variables and it is called the initialization list. This section is right before the opening curly brace of the constructor and right after the closing parentheses. It has the following syntax:

ClassName::ConstructorName() : variableNames(value to set) { }

The class name and constructor name stay the same. What is different is the colon which designates the start of the initialization list. The list consists of any class variables that you wish to set. Notice that parentheses are used instead of the equal sign to assign the value. If you have multiple values to set they should be separated by a comma: 

Rectangle::Rectangle() : length(0), width(0) { }
See what LearnCPP has to say about the initialization list

Initialize Data Within Header

While not recommended it is possible to write code and place it in the header file. You should only do this when the function your are writing consists of very small amounts of code. Initializing the attributes of the class in the header file with the constructor is one example of this that you will see quite often:

//Rectangle.h
#ifndef Rectangle
#define Rectangle

class Rectangle
{
    private:
        int length;
        int width;        
    public:
        int area();
        void setValues(int length, int width);
        Rectangle() : length() , width(0){} // Initialize attributes with constructor
};

NOTE: A couple of things to note here. First, if you put the code inside the header file it should not also be inside the .cpp file. This will cause a multiple definition error reported by the compiler. Second, since you are actually putting the function inside the header file you also need to make sure the braces are correct. In the above example all of the code is placed on one single line but it is valid function syntax.

See what XoaX.net has to say about constructors

Overloaded Constructors

Constructors in C++ can be overloaded just like any other function in C++. The power of this is that it allows you to construct your object in many different ways:

// Rectangle.h

class Rectangle
{
    private:
        int length;
        int width;
    public:
        int area();
        void setValues(int length, int width);
        Rectangle(); // Constructor
        Rectangle(int length, int width); // Overloaded constructor
};


// Rectangle.cpp
    
Rectangle::Rectangle() : length(1), width(1) { }
Rectangle::Rectangle(int len, int wid) : length(len), width(wid) { }

Creating Instances

Now you can construct the Rectangle objects in two different ways:

int main()
{
    Rectangle rectA; // Construct with default constructor
    Rectangle rectB(5, 6); // Construct Rectangle setting length to 5 and width to 6 return 0;
}

Default & Working Constructors

The constructor that has no arguments is called the default constructor. You may also see it called the no argument constructor. It is called the default constructor because if you don't create one the compiler will create one for you. You cannot use it or see it because it is tacked on to the compiled code but believe me it is there.

The working constructor is the constructor that takes on all of the possible settings at once. Here is another example which takes the Rectangle constructor out one more step:

// Rectangle.h
    
class Rectangle
{
    private:
        int length;
        int width;
    public:
        int area();
        void setValues(int length, int width);
        
        // Constructors
        Rectangle();
        Rectangle(int length);
        Rectangle(int length, int width); // overloaded constructor
};

Can you pick out the working constructor? It is the one that has the most arguments. 

// Rectangle.cpp

Rectangle::Rectangle() : length(1), width(1) { }
Rectangle::Rectangle(int len) :length(len), width(1) { }
Rectangle::Rectangle(int len, int wid) : length(len), width(wid) { }

One Code Path

I am one who believes that it is important to remove redundant code whenever possible. In the above examples the initialization of the length and width are repeated several different times. There is already a function written that sets the values. It is probably a real good idea to use it.

// Rectangle.cpp

Rectangle::Rectangle()
{
    this->setValues(1, 1);
}

Rectangle::Rectangle(int len)
{
    this->setValue(len, 1);
}

Rectangle::Rectangle(int len, int wid)
{
    this->setValues(len, wid);
}

As you can see this creates one code path. This way any time you need to change the way the object is constructed you only need to change the setValues function.


New To Version 11.0

Prior to the release of version 11 of the C++ compiler you could not have one constructor call another. In languages like Java the practice is to always have every constructor call the working constructor. I always thought this was a real limitation of the language. It is also very convenient for keeping one code path. This feature has been added in version 11 and is called delegating constructors.

Here is the example above this time using the version 11 convention of delegating constructors:

// Rectangle.cpp
    
Rectangle::Rectangle() :Rectangle(1,1) { }
Rectangle::Rectangle(int len) : Rectangle(len, 1) { }
Rectangle::Rectangle(int len, int wid) :length(len), width(wid) { }

Initialization Within Class Defintion

Another feature new to version 11 is the ability to initialize the attributes of the class inside the class definition. Notice that length and width are initialized where they are created:

// Rectangle.h

#ifndef RECTANGLE
#define RECTANGLE

class Rectangle
{
    private:
        int length = 1; //Initialize attributes in definition
        int width = 1;
    public:
        Rectangle();
        Rectangle(int len);
        Rectangle(int len, int wid);
        void setValues(int len, int wid);
        int area();
};

#endif

Prior to version 11 you could not put any initialization like this is the class definition. You were forced to do it in the constructor. While this is acceptable now you should exercise caution to make sure that your code is always going to be built on a compiler that adheres to the version 11 specification.


Uniform Initialization

Uniform initialization allows you to call the constructors using curly braces. You may also see this called brace initialization:

int main()
{
    Rectangle r{ 3, 5 };
    
    return 0;
}

The order here is important. The values in the braces should be in the same order as the parameters to the constructor. In the above example the number 3 would set the length and the number 5 would set the width


Constructor Notes

Initializing the object with the default constructor is a little odd. It seem logical that you could do this:

Rectangle rect(); // This is not correct

This seems logical because of the way you initialize an object with an overloaded constructor:

Rectangle rect(2, 3); // This is correct

However, the proper way to initialize an object with the default constructor is without parentheses:

Rectangle rect; // Also correct

More Calling Constructors

On the surface it seems that you should be able to recall a constructor like any other function:

Rectangle rect;
rect.Rectangle(5,6);

This is not allowed because the constructor gets called automatically when the object is created.

You can reinitialize an object with the constructor if you wish:

Rectangle rect;
rect = Rectangle(4,5);

What the above code does is create what is called an anonymous object. It is called anonymous because it is returning an object that has no name or reference. In this example, the object gets assigned back on top of the previously created Rectangle object.

You do not see this kind of thing used very often. If you think about the way that the Rectangle works you could simply call the setValues function which would reinitialize the object


Constructor Finally

Before the discussion on constructors is concluded it is important to point out one more thing. It was said that if you do not supply a default constructor the compiler will create one for you. This is true except in the case of overloaded constructors. If you have created any overloaded constructors you are then required to create the default constructor. Failure to do so will result in a compiler error.

Using Constructors

Destructors

A destructor works the exact opposite way of a constructor. Where a constructor is used to initialize an object, a destructor is used to clean up an object. A constructor is called automatically when an object gets created; a destructor gets called when an object is being removed from memory. An object gets removed from memory when the program exits or when an object that has been created dynamically is deleted from memory using the delete operator. The concept of dynamic memory will be covered in a future lecture. The thing to know at this point is that the destructor will get called automatically when an object is being removed from memory.

The constructor has the same properties as the constructor:

  1. *It must be declared in public
  2. It has the same name as the class
  3. It does not have a return type not even void

*It is possible to have a private destructor but when you do this there is no way for you to destruct the object publicly. This will be covered in more detail in the sections on composition and inheritance.

What differentiates the destructor from the constructor is that the destructor is prefixed with a ~ (tilde):

#ifndef RECTANGLE
#define RECTANGLE
class Rectangle
{
    private:
        int length;
        int width;
    public:
        Rectangle();
        void setValues();
        int area();
        // Rectangle destructor
        ~Rectangle();
};

#ifndef

The body of the destructor then is easy to create:

Rectangle::~Rectangle()
{
    // Clean up object here
}

It is hard to imagine what you would put in a destructor for the Rectangle class. A better analogy might be the idea of an Elevator class. When you are done using the elevator you might want the elevator to return to the ground floor. This code, to return to ground floor, would go inside the destructor.

Introduction

To understand the need for a header file you must first have an understanding how the build process works in C++. You write source code in plain text and save it to a file with the extension .cpp. This file is then ran through the preprocessor which handles anything that has # in front of it. For instance #include<iostream> which opens and includes the contents of the iostream header into the source code. It will also handle macros and constants created with #define.  From there the .cpp file is ran through the compiler which among many other things checks the source code file for syntax errors. If there are no errors the compiler creates assembly language code for the processor in which it is being compiler for. This assembly language code is output into a file with the extension .obj (object code).  At this point the object code is ran through the linker which puts this code together with any other object code along with any of the functions used from the standard library. The output from this linker is an executable file with the extension .exe.

This process is depicted in the following diagram:

The above process only depicts the compilation of one file. The truth is that most programming projects contain many files. The thing to keep in mind with this is that the compiler treats each one of the files separately.  It has no way of knowing that a function you are calling in one file appears in another. The compiler though is good with this as long as it can see a prototype of what the function consists of. It is the linkers job to insure that all of the prototyped functions actually exist. If  there is a prototype for a function in the a file but the function body does not exist, the compiler will still pass the file through to the linker. If the linker cannot find the actual body of the function in any of the object files it will create what is called and "unresolved external" error. 

With this in mind then we can say that the compiler needs to see a definition of anything before you can use it. These definitions include prototypes, classes, and structs. 

Now consider you have hundreds of files in your project and in about half of them you need to use a  certain class. Knowing that the compiler needs to see this class definition does it make sense to include this definition 50 times or place the class definition in a header file and include the header file 50 times? 

Think of the maintenance here. If the class changes you are going to have make changes in 50 places instead of one. 

When the C++ language was first created all header files had the extension .h. To include the iostream header for instance one would write it like so

#include <iostream.h>
int main()
{
    return 0;
}

Since the inception of namespaces and in particular namespace std all of the header files that are part of the standard library have the .h dropped from the end.

Creating Your Own Header

Now when you write classes you are more than likely going to want to create a header file to put the class definition in. Of course you don't have to do this but remember if you want to use the class definition is several .cpp files you are going to be duplicating the definition many times.

When your create a header file you will give it the .h extension. Here is an example of the Rectangle class

//Rectangle.h
class Rectangle
{
private:
   int length;
   int width;

public:
   void setValues(int length, int width);
   int area();
   Rectangle();
   Rectangle(int length, int width);
};

The only thing new here is that this code would go inside the file Rectangle.h. You would still have to put the code for the functions that are prototyped inside a file named Rectangle.cpp

//Rectangle.cpp
 #include <iostream>
 #include "Rectangle.h"
Rectangle::Rectangle()
{
	Rectangle(1, 1);
}
Rectangle::Rectangle(int length, int width)
{
	this->setValues(length, width);
}
void Rectangle::setValues(int length, int width)
{
	this->length = length;
	this->width = width;
}
int Rectangle::area()
{
	return this->length * this->width;
}

Quotes or Brackets

Looking at the above code you may be asking yourself why are there brackets around iostream but quotes around Rectangle?

The answer is that most IDE's these days have an include path that gets set when you install the compiler. What is in the include path are the headers that go along with the standard library. Anything that is in this path should have the brackets <> around it and anything that is in the local folder should have the quotes around it "". 

Now it is possible to add your header to the include path or add more folders to the include path. Typically speaking though you want the header files to be in the same location as the code they define. In the case of the Rectangle we want the header in the same location as the project. 

Of course if you code long enough you may develop some functions that you will use across many projects. In cases like this you would create a library and add the header that goes with the code to the include path. This way it can be referenced by any new project created. 

One last thing about including header files and that is that it is possible to specify a complete path in the include string:

#include "c:\\myheaders\\myHeader.h"

Notice the use of the double backslash \\? Remember the backslash indicates an escape character. Remember "\n"? Without the second backslash the compiler would look at the next character as an escape character.

As a word of caution, if you are going to include the path then you should avoid absolute paths and use a relative path instead:

#include "..\\myheaders\\myheader.h"

Including Definitions Once

You may think that including a header multiple times in you .cpp file is something that you can easily prevent. The fact is that it is not that easy. Most header files include several other header files. If you think about this then anytime you include more than one header file in your code you are in danger of adding the header twice. 

Fortunately, you can use some statements from the preprocessor to guard against your header being included multiple times. Here is an example:

 

#ifndef SOMELABEL
#define SOMELABEL

// Code for header goes here

#endif

  • ifndef statements means "If Not Defined". This checks to see if SOMELABEL has been defined before
  • define - defines SOMELABEL to the system
  • endif - ends the ifndef statement block

When you compile source code that has a header included in it, the preprocessor will open the header to include it's contents into the source file. If the ifndef statement yields a false result then the next line is executed and SOMELABEL gets defined to the compiler. Then all of the definitions are included into the source.

If the ifndef yields a true result meaning that SOMELABEL had been defined before (This could only happen if the header was included more than once in a source code file) the definitions are skipped and not inserted into the source code. This mechanism prevents definitions from appearing more than once in your code.

#pragma once

If you are using a later version of Visual Studio and have included header files into your project then you probably have noticed the statement; 
#pragma once

at the top of the page. This piece of code will do everything that the header guards will do. The biggest problem with it is that it is not supported by all compilers. Although, it is getting to be more common. I suggest you stick with the old way of doing header guards. This way you are assured that it will work. At least until this become a little more popular.

Including Other Headers

When you are writing code it is important that you make your files as modular as possible. This means that if you are using something that needs an include, you include it. This seems pretty logical but it is possible to get things included simply by the order in which they are placed inside the .cpp file. Here is an example of the Rectangle class header again. This time the string forExample has been added simply for example purposes: 
#ifndef RECTANGLE
#define RECTANGLE

class Rectangle
{
private:
	int length;
	int width;
	string forExample;

public:
	void setValues(int length, int width);
	int area();
	Rectangle();
	Rectangle(int length, int width);
};

#endif

This all looks fine until you try to compile it. The compiler is going to give an errors saying that it does not understand what a string is. You can fix this error by the order in which you place things in main: 

#include <string>

using namespace std;

#include "Rectangle.h"

This works because the compiler includes string and resolves the namespace before opening Rectangle.h. You should NOT rely on this. If your class gets put into another project you cannot guarantee that someone will put things in the proper order. You must assure that no matter how your header and code is used it will work. The answer is to include the header and resolve the namespace in the header file:

#include <string>  //prototypes for the string

#ifndef RECTANGLE
#define RECTANGLE

using std::string;  //Resolve the namespace

class Rectangle
{
private:
	int length;
	int width;
	string forExample;

public:
	void setValues(int length, int width);
	int area();
	Rectangle();
	Rectangle(int length, int width);
};

#endif

Notice that the include is placed before the header guard? Prior to version 11 of the compiler you would get a warning about nesting your namespaces. While this is not a problem now you may want to follow suit for backward compatibility sake

using namespace

Inside a header file you should never place using namespace std in it. When you do this you are now forcing any .cpp file that includes your header to the std namespace. This not not good thing if you are using multiple namespace in a project. Instead you want to resolve everything individually. In the case of the string you could do as the above example and place using std::string at the top of the file. Or you could simply prefix every declaration with std::

#include <string>   //prototypes for the string

#ifndef RECTANGLE
#define RECTANGLE



class Rectangle
{
private:
	int length;
	int width;
	std::string forExample;  //Resolve the namesapce

public:
	void setValues(int length, int width);
	int area();
	Rectangle();
	Rectangle(int length, int width);
};

#endif

Whichever method you use be consistent and remember, you need to do this for any code you are using from the standard library.

Introduction

At this point in time it is probably a good idea to talk about debugging and go over your options. First off you should know that the debugger does not catch compile errors. This of course will be done by the compiler. What the debugger does is to give you the tools needed to find logic errors in your code. As you write more and more complex code the chances of logic errors creeping in grow also. having the ability to walk through your code line by line and see what is going on at any instance is a real benefit

Most Integrated Development Environments (IDE) these days have a debugger built into them. In the section we are going to be looking the Microsoft debugger because I believe it is one of the most mature tools out there. Don't worry though if you are using a different tool most debuggers work the same way.

Debug Or Release?

You may not know it but there are actually two ways you can build your project. One is for debug and the other is for release. By default Visual Studio creates a new project for debugging. This is fine for class work and things of this nature but if you were to build a product that you were selling you would most definitely want to build it for release.

So what is the difference you may ask? When you build a project for debugging the debug symbols are added to the project. These symbols are found in a file with the extension pdb. Here is a description provided by Microsoft:

"A program database (.pdb) file, also called a symbol file, maps the identifiers that you create in source files for classes, methods, and other code to the identifiers that are used in the compiled executables of your project. The .pdb file also maps the statements in the source code to the execution instructions in the executables. The debugger uses this information to determine two key pieces of information: the source file and line number that are displayed in the Visual Studio IDE and the location in the executable to stop at when you set a breakpoint. A symbol file also contains the original location of the source files, and optionally, the location of a source server where the source files can be retrieved from".

"When you debug a project in the Visual Studio IDE, the debugger knows the default location for the .pdb and source files for your code. If you want to debug code outside your project source code, such as the Windows or third-party code your project calls, you have to specify the location of the .pdb (and optionally, the source files of the external code) and those files need to exactly match the build of the executables".

"Before Visual Studio 2012, when you debugged managed code on a remote device you needed to put the symbol files on the remote machine. This is no longer the case. All symbol files must be located on the local machine or in a location specified in the Tools / Options / Debugging / Symbols page".

When you are finished developing your product and are ready to ship it then you should change the build type from debug to release. If you ship with the debug symbols in the code who ever has your product can then go through and see your source code. 

Basics

The best starting place here is to identify the buttons that are part used for debugging.

 Icon Name  Keystroke  Description 
   Step Into F11  If you are stopped at a function the step int button allows you to step into into. 
   Step Over  F10  The step over button allows you to move past a function. The function will still be called.
   Step Out   Shift F11  Allows you to immediately step out of a function. 
   Stop Debugging  Shift F5  Immediately stops the debugger and exits the code. 

 

It should be noted that the debug toolbar is not displayed until you start the debugger. If after starting the debugger you still do not see the debug toolbar then simply go View / Toolbars / Debug and make sure there is a check next to Debug. 

 To start a debug session you are probably going to want to set a break point. A break point is any valid line of code where you want the debugger to stop executing at. If you do not set a break point then your code will simply run to completion and exit. A break point can be set in a couple of different ways

  1. Right click on the line of code where you want the execution to stop at then select  Breakpoint / Insert Breakpoint.
  2. Click in the grey bar to the left of the code window on the line of code you want the execution to stop at.

With either of these methods you should see a red dot appear to the left of the line you want to stop at.

Once you have breakpoint established you can now start the debugger. You can do this a couple of different ways

  1. Through the menu by selecting Debug / Start Debugging
  2. By selecting the green arrow head icon
  3. By pressing F5 on your keyboard. 

You can set as many breakpoints as necessary. Many times you will find it useful to set a breakpoint before and after a loop. This way you do not have step through every iteration. 

Run To Cursor

Another method to step over loops or to simply execute to a spot without using a break point is to simply right click on a line and then select  "Run To Cursor"

Viewing Variables

With the newer debuggers you can view what a variable contains simply by rolling the cursor over the variable at any given spot. The value of the variable will show in a small popup window. 

Another way to view the value of a variable is by creating a watch. A watch shows the variable in a window called the watch window. In Visual Studio you can set a watch simply by selecting Debug/QuickWatch from the menu or by pressing shift+F9. Either of these methods will open up the QuickWatch dialog box:

In the expression text field you simply put the variable name you wish to watch and select and select "Add Watch". This will open the watch window typically at the bottom of the screen and show the variable you added along with it's value:

Anytime the contents of the variable change the Value will be displayed in red.  Another feature of the watch window is that you can change what the variable holds simply by double clicking the value in the watch window and typing in the value you want the variable to have. 

NOTE: In the case of a string you need to expand the variable and change individual characters
