Introduction

There are basically two types of errors that can be generated by your program they are

  1. Compile Time Errors
  2. Runtime Errors

    3. Compile Time errors should be the easy ones to fix because they are caught by the compiler. The compiler spits out a nice error message that includes the line number that lets you know where the error was encountered and description of of what the error is. The trick is to understand the message the compiler generates.

    Take a look at the following code. How many errors can you spot?

    #include <iostream>
using namespace std;

int main(String agrs[])
{

num = 32;

int var;
    cout << "var = " + var << endl;
    cout << "New = " << num;

    return true;
}

    Do you think that you have found all of them? Let's take a look at what the compiler reported

The above is a snap shot taken from Visual Studio. As you can see it shows there are 7 errors. What you should notice is that the majority of the errors reported are located on lines 5 and 6. So we can say that this is an incomplete list of errors. In reality, some errors can mask the presence of other errors. If we fix the errors on lines 5 and 6 then the compiler will show us other errors. Have you ever been fixing errors and finally got it down to one and when you fix that error, another dozen or so pop up?

The error descriptions in this case do not seem to be very helpful but if we examine the list we should note that error 1 on line 5 says undeclared identifier. There is actually two things wrong here. First, the String is an undeclared identifier because we have not included <string> which is part of the std namespace. The other thing is the main function does not take an array of strings as its argument. Lets fix that and see what happens:

Types of Compiler Errors

There are basically four types of compiler errors:

  1. Structural
  2. Typographical
  3. "Real"
  4. Type Errors

Structural Errors:

Structural errors are things like missing or unmatched parentheses, missing or unmatched quotes, and missing or unmatched braces. The first step to fixing compile time errors in your program should be to fix the structural errors first.

Three Steps To Avoiding Structural Errors

  1. Know what goes where - include statements must appear first, formal agruments must go inside parentheses, all statements must appear inside a function
  2. Write structural statements in one piece - add the closing brace immediately after adding the opening brace etc.
  3. Use a consistent and easily checked indenting style

Real Syntax Errors

The C++ compile divides your code into chunks. Having done this, it checks to make sure all of your chunks are in the right place. These are called "Real Syntax Errors". An example of a "Real Syntax" error would be using a keyword in an inappropriate place:

int double a;   //which type is it?
int a = 10, int b = 10;  //Can't declare types twice

Another example would be the omission of a necessary keyword

// Each formal parameter must have a type
void foo(int x, y)
{
}

The only real solution to avoiding "Real" syntax errors is "MEMORIZATION" and paying attention to the details.

Typographical Errors

Typographical errors are ones you get when you misspell a word, forget a semicolon, or use the wrong case. Fortunately in this day and age code completion and intelli-sense make avoiding these errors easy. You should learn to use your editors code completion and intelli-sense functionality

Type Errors

Type errors occur when you try to store a value of one type into a variable that is meant to hold another type. Here are three rules to help you avoid type errors:

  1. Store little values in bigger data types. You can store a char in an int but you cannot store an int in a char variable without casting
  2. Store less precise values in more precise data types. You can store a float in a double variable but you cannot store a double in a float without losing precision.
  3. Store similar things. A bool data type cannot hold a numeric value or an object

Exception Handling In C++

In the world of the 'C' programming language, error handling was done with completion codes. A completion code is a value that gets returned from a function. The programmer would call the function and then check the value that was returned. It was a typical convention for a function to return a value of 0 (zero) if the function completed successfully and non-zero if the function returned and error. It was also typical to create constants for the completion code. This made the code much easier to read and understand

Here is an example

const int FILE_OPEN_ERROR = 1;
const in FILE_NOT_FOUND = 2;
const int FILE_READ_ERROR = 3;
const int FILE_WRITE_ERROR = 4;
const int FILE_CLOSE_ERROR = 5;

void someFileThing()
{
int status;

    status = openFile();
    if(status == FILE_OPEN_ERROR)
        //Handle file open error
    status = writeFile();
    if(status == FILE_WRITE_ERROR)
        //Handle file write error
    status = closeFile();
    if(status == FILE_CLOSE_ERROR)
        //Handle file close error

    return;
}

For the sake of accuracy there was no const in the C programming language. This was all done using the preprocessor: 

#define FILE_OPEN_ERROR 0

In addition, you would see quite a few switch statements that would switch on status then branch to the appropriate error

The above example is very simplistic but it is indicative of how error handling was done. What you should notice is that the program is so consumed with concentrating on errors that the purpose of the code gets a little cloudy.

A better solution is with the error handling triad try-throw-catch that was added to the C++ language. The following shows how the try catch can be used to separate the code that works from the exceptional code:


void someFileThing()
{
 
    try
    {
        openFile();   
        writeFile();   
        closeFile();
    }  

    return;
}
// If an error occurs in any of the above, handle it here

As you can see this makes the code much cleaner because it separates the working code from anything exceptional.

Try Catch

C++ supplies a mechanism like the above that allows you to separate the error handling code from the code that is performing the task. This mechanism is in the form of a try / catch block. The general syntax for the try / catch block in C++ is:

try
{
    // Code that could generate an exception goes here
}
catch(SomeException)
{
    // Handle the exceptional situation in this block
}

The above assumes that the code within the try block throws an exception. The main idea here is that you can place a try block around a function or other code that could possibly throw an exception. When an exception is thrown, it branches to a particular catch block which handles the exceptional situation. 

The catch block looks like a function definition with one argument. It is really not a function but it does work similarly. The catch block can only have one argument. If you need to catch several different exceptions you need to place a separate catch block for each type that can be thrown.

Throwing An Exception

In C++ you can use the throw to throw an exception. This can be done from anywhere in the code. The following is an example of how to throw an exception from within a try block.

int main()
{
int noodles;
int meatballs = 5;

    try
    {
        cout << "How many ounces of noodles? " << endl;
        cin >> noodles;
        
        if(noodles == 0)
            throw meatballs;

        cout << "Spaghetti and meatballs are really good together" << endl;
    }
    catch(int mb)
    {
        cout << mb << " Meatballs with no spaghetti. Time to make more noodles" << endl;
    }

    return 0;
}

The above code asks the user for an amount of noodles. If noodles are 0 then the code throws meatballs. You should note that meatballs is an integer. This means that the try block throws an int as an argument to the exception.

As was stated earlier you can throw as many exceptions as you need but you must supply a catch block for each type that can be thrown. The following example catches a String and a double

int main()
{
    try
    {
        invest();
        cout << "You have made a wise investment " << endl;
    }
    catch(string msg)
    {
        cout << msg.c_str() << endl;
        exit(1);
    }
    catch(double amt)
    {
        cout << "The unbelievable low price is only " << amt << endl;
        return 1;
    }
    cout << "I am not sure what you mean " << endl;
}
void invest()
{
char ch;
   
    cout << "Would you like to by beach front property in Nevada? (y/n)" << endl;
    cin  >> ch;
    if(toupper(ch) == 'Y')
        throw 'You have just been swindled";
    else if (toupper(ch) == 'N')
        throw 1095535.0;

}

Catch All

If you don't know what type of exception a function will throw or maybe you are just a little lazy and don't want to catch each different type, or maybe you would just like to have one catch block that will catch any error no matter what type it may be. Well, you are in luck, you can use ellipses as the catch block argument and it will catch any error that is thrown. Of course, you have no way of knowing exactly what the type is but there are times when you may not care:

try
{
    //Code that generates the exception
}
catch(...)
{
    // Handle any exception here
}

The following example demonstrates the use of the elipses

int main()
{
    try
    {
        invest();
        cout << "You have made a wise investment " << endl;
    }
    catch(string msg)
    {
        cout << msg.c_str() << endl;
        exit(1);
    }
    catch(...)
    {
        cout << "The unbelievable low price is only " << amt << endl;
        return 1;
    }
    cout << "I am not sure what you mean " << endl;
}
void invest()
{
char ch;
   
    cout << "Would you like to by beach front property in Nevada? (y/n)" << endl;
    cin  >> ch;
    if(toupper(ch) == 'Y')
        throw 'You have just been swindled";
    else if (toupper(ch) == 'N')

}

In the above example the elipses is used to catch the double. You might consider using the elipses at the end of a long list of catch blocks as a safety to catch any error that you did not plan for.

If your function threw an error that was not part of the throw list the function unexpected would be called. The unexpected function in turn would call the terminate function which would force your program to exit.

One of the nice things about the unexpected function is that you could call set_unexpected and provide a pointer to a void function as a parameter. This function would now be called instead of unexpected.

If you have written a function that throws an error that is not caught properly the terminate function will be called. This could include either not having a catch block at all or simply not having a catch block wit the appropriate data type as a parameter

The exception list has been known to be problematic. For this reason it has been deprecated in the version 11 of the C++ standard. You can still use it but you will probably get an ugly compiler warning.

Built In Errors

In C++ you can throw any data type as an error. This includes complex types. C++ has many built in classes that you can use to catch a variety of exceptions. All built in exceptions are derived from exception. In order to use one of these classes in your code you must first include exception. The following is a brief hierarchy of t he exception classes in C++

Many of these exception classes are self explanatory. If you need further explanation you should go to the online C++ reference

The base class exception has a virtual function called what that provides a CString that contains information about the exception. All of the derived classes override this exception to provide their own error message.

The following example shows how to use the built in error bad_alloc to test allocation with the new operator to create space for an array.

#include <iostream>
#include <exception>

using namespace std;

int main()
{
    char *p1 = 0;
    try
    {
        p1 = new char[1000000000000];
        cout << "p1 allocated successfully" << endl;
    }
    catch (bad_alloc& e)
    {
        cout << "Failed to allocate p1" << endl;
        cout << e.what() << endl;
    }
    
    return 0;
}

Creating Custom Error Objects

There may be times when the built in exception classes do not have any descriptive details to handle the exception that can be thrown from you class. In cases like these you may want to create your own exception class. The following code demonstrates the creation of a custom exception class that is derived from exception 

class SomeMath
{
private:
	double num1, num2;
public:
	SomeMath() : num1(1.0), num2(1.0) {}
	SomeMath(double n1, double n2) : num1(n1), num2(n2) {}
	double divide();
};

class MathException : public exception
{
private:
	string message;
public:
	MathException(string msg) : message(msg) {}
	virtual const char *what() { return message.c_str(); }
};
double SomeMath::divide()
{
	if (this->num2 == 0)
		throw MathException("Cannot Divide by 0");
}

The following main function tests the custom exception class

int main()
{
	SomeMath sm(5.2, 0);
	try
	{
		double quotient = sm.divide();
		cout << "The quotient is " << quotient << endl;
	}
	catch (MathException ex)
	{
		cout << ex.what() << endl;
	}

	return 0;
}

The above code shows a typical exception handling solution. If the exception happens in the class then it is simply thrown. The try - catch block is wrapped around the code in main where the function that could generate the exception is called. A mistake is to put the try-catch block in the class. Doing this does not give the person using the class the flexibility to handle the exception in their own way.

Re-throwing an error

There may be times when you do not want to handle an error where it is caught. This might be the case in a class that calls a function from another class. In situations like this you can simply re-throw the error. This is done simply by placing the keyword throw when the exception is caught

try
{
     // Try code here
}
catch(SomeError)
{
     throw; // re-throw the same exception up  the chain
}

You can also throw a different exceptionup the chain

catch(SomeError)
{
     throw newExceptionUp; //throw a different exception up  the chain
}
