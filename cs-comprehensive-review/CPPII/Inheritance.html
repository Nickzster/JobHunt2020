Introduction

The development of Object Oriented programming has greatly improved software development. In the early days of software there really weren't any formal techniques used for solving software solutions. At one time in the history of Computer Science the development of hardware far outpaced that of software development practices. The result of this was that there was a bunch of poorly engineered software. So much in fact that this time in history of software development has been termed the software crisis.

Up to this point we have been talking about how we can bundle data along with functions that operate on that data to create an efficient object. What we haven't discussed to this point is program organization through different object relationships.

We need to have a system of organization in software development because we humans are complexity challenged. Many people may not realize it but software solutions tend to hang around for many years. One of the reasons for this is the high cost of development. Once a program is up and running it is much cheaper to maintain it than develop a new system. Highly organized code is much easier to modify and maintain than code that is thrown together is a sloppy fashion.

The first breakthrough in software organization was through functional decomposition also know as step wise refinement. With this methodology problems were broken down into functions. Each function then solved a problem. If the function had a sub problem another function would be created and called. This type of programming is still used in many places today. In fact this is primarily what was done in the level one class. Of course as we have seen the problem with this type of organization is that it does not allow a means to bundle data and operations. It also is not extremely efficient in terms of code reuse.

OOP Types Of Organization

There are really three main types of organization found in Object Oriented programming

  1. Communities - Also called a "uses" relationship
  2. Composition - Also called a "hasa" relationship
  3. Inheritance - Also known as an "isa" relatioship

Communities

In community based programming objects offer their services and request the services of other objects. You can think of this as a client / server type relationship where one object acts as the director and sends and receives requests from other objects. In other words an object "uses" the services of another object.

Very large programs have been built using this type of organization. The C# and Visual Basic languages employ this type of model. A form gets created and filled with components such as buttons and text boxes. Each of these components communicates with other by requesting that an action be performed to change one of their attributes

Composition

You have been using composition for sometime now. This is nothing more than one object being composed from at least one other. In other words an instance of another object is declared inside the data section of another class. Here is a simple example:

struct Person
{
string first;
string last;
string address;
};
class People
{
private:
   vector<Person> people;
public:
    void addPerson(string first, string last, string address);
    string getFirst();
    string getLast();
    string getAddress();
};

In this example the People class is composed of a vector. In other words you can say that the People class "hasa" vector within in.

Inheritance

Inheritance allows you to create a specialized version of an object. A crazy example of this might be that of a flying car. If you were to tackle such a project you probably wouldn't first go out and try to build a car. Instead you would probably start with a car the has already been manufacture and then attempt to figure out how to make it fly. As you can see the general case is a simple car. The specialized case is making it fly.

This kind of relationship really promotes the idea of code reuse. Before OOP you would simply have to build the car first or borrow parts from an existing car to build your own. The idea here is that by simply using inheritance you get all of the functionality of one object into another.

The class that is the general case is typically called the base class. The specialized class is typically called the derived class. You can think of this in many different ways like parent / child of super / sub class. Using inheritance then allows your class to become a specialized version of another class. We can say then that a flying car "isa" car.

Inheritance In C++

Inheritance allows you to create classes which are derived from other classes. This means that they automatically include some of their parent's members along with their own. One of the classic examples of inheritance is that of shapes. Most shapes as we will see have common features. These common features would be extracted and put in the parent class. The idea of extracting the common features is called generalization because you are defining the attributes that will be the general case for all derived classes. In the case of shapes the general class is going to be called Polygon because a rectangle "isa" Polygon and a Triangle "isa" polygon also. The common traits are that shapes can be described in terms of two sides, the height and base. Through generalization then we see that the common properties should be grouped and put into a class of their own.

Subclasses

From polygon, Rectangle and Triangle are derived. Rectangle and Triangle then are both children of Polygon. This parent / child relationship is also called super class and subclass. That is, Rectangle and Triangle are subclasses of polygon.

You should note that anything the parent can do the child can do also. This is accomplished through the principle of substitutability . That is to say that a Rectangle or a Triangle is also a Polygon but not the other way around. A polygon is not necessarily a Triangle or a Rectangle.

Creating a derived class in C++ is quite easy. You simply use a colon to designate the parent class:

In this example derived_class_name is publicly derived from base_class_name. Or to put it another way, base_class_name is the parent of derived_class_name.

Classes that are derived from other classes inherit all of the accessible member functions and variables of the base class. That means if we have a base class which includes a member A and we derive another class from A with a member called B , the derived class will contain both members A and B .

Protected

As was discussed earlier any data that is placed in the private section of a class can only be accessed by members of that class. This is still true in the case of inheritance. Class Rectangle cannot access the private members width and height of the Polygon class. This is where the protected section comes in to play. Protected is very similar to private in that it restricts access to anything outside of the class. Derived classes though are allowed access to the members of this section.  The classes Rectangle and Triangle each contain members inherited from Polygon (width, height and set_values()).  Since we want width and height to be accessible from members of the derived classes Rectangle and Triangle we have use protected access instead of private.

The different access types can be summarized in the following way

Access public protected private
Members of the same class yes yes yes
Members of derived classes yes yes no
Non members yes no no

Where "non members" represent any access from outside the class, such as from main() , another class or from a function.

In the example, the members inherited by Rectangle and Triangle have the same access permissions as they had in their base class Polygon :

Member  Access 
 Polygon::width protected
 Rectangle::width protected
Polygon::set_values() public
 Rectangle::set_values() public

Inheritance Relationships

We have seen that each class can have sections with different access specifiers. The inheritance relationship can specify access also:

class Rectangle::public Polygon
{
};

The above specifies that the Rectangle class is publicly derived from the Polygon class. As with the class access specifiers the inheritance can also be public, private or protected. The following table explains the effect each access specifier has on the inheritance relationship:

Access public protected private
Members of the same class yes yes yes
Members of derived classes yes yes no
Non members yes no no

Where "non members" represent any access from outside the class, such as from main() , another class or from a function.

In the example, the members inherited by Rectangle and Triangle have the same access permissions as they had in their base class Polygon :

Access  Description
public The derived class will inherit all the members from the base  class maintaining their access levels
protected All public members in base are inherited as protected in derived class.
private All public members in base are inherited as private in derived class.

The public keyword after the colon ( : ) denotes the minimum access level for all the members inherited from the class that follows it (in this case Polygon ). Since public is the most accessible level, by specifying this keyword the derived class will inherit all the members with the same levels they had in the base class.

If we specify a more restrictive access level like protected , all public members of the base class are inherited as protected in the derived class. Whereas if we specify the most restricting of all access levels: private , all the base class members are inherited as private.

For example, if daughter was a class derived from mother that we defined as: 

class Rectangle : protected Polygon{...}

This would set protected as the maximum access level for the members of Rectangle that it inherited from Polygon . That is, all members that were public in Polygon would become protected in Rectangle. Of course, this would not restrict Rectangle to declare its own public members. The maximum access level is only set for the members inherited from Polygon.

What is Inherited?

In principle, a derived class inherits every member of a base class except:

  • constructor and destructor
  • operator=() members
  • friends

 

Although the constructors and destructors of the base class are not inherited themselves, its default constructor (i.e., its constructor with no parameters) and its destructor are always called when a new object of a derived class is created or destroyed.

If the base class has no default constructor or if an overloaded constructor is called when a new derived object is created, you can specify it in each constructor definition of the derived class: 

derivedContructor(parameters) :    baseConstructor(parameters){...}

Multiple Inheritance

Up front I am going to tell you that I am not a big proponent of multiple inheritance. It kind of violates most of the principles of object oriented programming. The C++ language came along before any real OOP principles were agreed upon. if you look at any OOP language that has come along since C++ (Java, C#) you will notice that multiple inheritance is not part of the language. From my point of view, it also can lead to a bit of confusion and cluttered code. I am not one who would tell you to never use it. If you find yourself in a situation where using it leads to an elegant solution then you should by all means use it.

Multiple inheritance in C++ is done by simply separating the different base classes with commas in the derived class declaration. For example, if we had a specific class to print on screen ( Output ) and we wanted our classes Rectangle and Triangle to also inherit its members in addition to those of Polygon we could write: 

class Rectangle : public Polygon, public Output;
 class Triangle : public Polygon, public Output;

Introduction

The word Polymorphism means that one object can have many different meanings. To understand what this means to us in programming let's consider shapes again. The examples seen so far have a base class called Polygon and two classes Rectangle and Triangle derived from Polygon. When we create classes we want to generalize things. This means that we want to remove the redundant (common) code and place it in the base class. This keeps us from having a bunch of duplicate code.

In Polygon we want to place a function called center which will draw the object in the center of the screen. The problem here is that each shape needs to be drawn differently. This means that each shape must have it's own draw function. This can be depicted as follows

class Polygon
{
protected:
	int width, height;
public:
	
	void center();
};
void Polygon::center()
{
	...
		draw();
	...
}
class Rectangle : public Polygon
{
public:
	void draw();
};
class Triangle : public Polygon
{
public:
	void draw();
};

When you create an instance of Rectangle or Triangle and want to center the object on the screen you would call the center function that is defined in Polygon. The center function in turns calls the draw function. The question is how does it know which draw function to call? Does it call the draw function in Rectangle or the draw function in Triangle? Also, what happens if later on a new shape is created say Circle? The center function was created long before the Circle class was thought of. This is where polymorphism comes in to play

Polymorphism is achieved through the use of virtual and pure virtual functions. Before getting into that we need to look a few other concepts first.

Overloading Revisited

To review, an overloaded function is multiple functions that have same name but differ by their argument list in some way. It could be the number of parameters, the data types of the parameters or both. One thing to remember is that an overloaded function cannot simply have a different return type. It must differ by the parameter list.

The following example shows two functions called addem. One takes two int parameters and the other takes three int parameters

#include 

using namespace std; 

int main()
{
int x,y,z; 

   cout << "Enter 3 integers " << endl;
   cin >> x >> y >> z;

   cout << "The sum of the first two is " << addEm(x, y) << endl;
   cout << "The sum of all Three is " << addEm(x, y, z) << endl;  

}

int addEm(int a, int b)
{
    return a + b;
}
int addEm(int a, int b, int c)
{
   return addEm(addEm(a,b), c);
}

Because of the "isa" relationship it is possible to have overloaded functions where some appear in the base class and others appear in the derived class.

class father
{
public:
    void saySomething();    // Base class saySomething function
};
class son : public father
{
public:
    void saySomething(string sentence);
};

In the above example both the base and derived classes have a function called saySomething. The only difference between the two is that they have different argument lists.

Overridden Functions

Overridden functions are similar to overloaded functions. The differences are that they both have the same parameter list and overridden functions cannot appear in the same class. They must be in base / derived classes. When you override a function you are redefining it's behavior in the derived class. This allows you to write a function that has the same name but is specific to the specialize version of the class.

class father
{
public:
    void saySomething();   // Base class saySomething
};
class son : public father
{
public:
    void saySomething(); // redefined in the derived class
};

In the above example the functions defined in the base and derived classes have the same name and signatures. The function in the derived class overrides the function in the base class. In other words it redefines the function saySomething specifically for the son class.

When a function is overridden the version that is in the created object is the one that will execute:

son s;
s.saySomething();  // Calls the derived class function

Calling the Parent

To call the base class version of the function from the derived class you need to use the scope resolution operator to specify which class the function is defined in. 

class son : public father
{
public:
    void saySomething();
    void speak();
};
void son::speak()
{
    //Call the son class version of saySomething
    this->saySomething();

    //Call the father class version of saySomething
    father::saySomething();
}

How To Determine

Overloaded functions - Functions that have the same name but different parameter lists. They can be found together in the derived or the base class. Or, they can be separated one in base class and the other in the derived class.

Overridden function are functions that have same name and parameter list. one function is in the base class, the other in the derived.

Virtual Functions

The introduction to this section discussed the Shape class and the problem with writing a center function in the base class and calling the correct draw function in a derived class. This problem is solved by using virtual functions. A virtual function is a class member function declared using the "virtual" keyword. Because of the "isa" relationship a pointer to a derived class object can be assigned to a base class instance. A virtual function then can be called through the base class pointer. If the function is virtual and overridden in the derived class the correct function will be called based on what the base class pointer "really" points at. 

Shape *p= new Line(0.1, 0.1, blue,0.4,0.4);
p->draw();

The draw() function for a Line is called, not the draw() function for Shape. This is called polymorphism which gives a type the ability to appear and behave as another type.

Now that you have an idea as to what a virtual function is you may be asking yourself how does this really help me solve the problem of centering a shape? The answer is that you would create the center function which will call a virtual function called draw. This way each shape that is created will have to implement it’s own draw function.;Let’s look at some example code:

Pure Virtual Functions

Before talking about pure virtual functions there is a some terminology that needs to be understood

  • Abstract Classes - Abstract classes are classes that you cannot create an instance of. This means that the are used for inheritance purposes only.
  • Concrete Classes - Concrete classes are class that you can create instances of. These are the types of classes you have been using up to this point

Pure virtual functions work similar to virtual functions except they make a class abstract. Because they are abstract they are used for inheritance only. When you create a pure virtual function you do not create a function body to go with it. In other words you are simply creating a prototype in the class definition. This means that the derived class MUST implement the function. If the derived class does not implement the function then that class becomes abstract also and the compiler will generate an error.

Pure virtual functions force the user of the class to implement the function. The reasoning is this, when you created the shape class you knew that whoever used it was going to need to create their own version of the draw function because the Shape class has no idea what shapes will be added and how to draw them. This is like a contract that says it is OK for you to derive your classes from me but there is a function that you must implement to do it. When declaring a pure virtual function you create a prototype and then set it equal to NULL (0):

virtual void draw() = 0;

Placing the 0 says that this is a pure virtual function. Notice also that the Shape class does not create a body for this function. If you were to try to create an instance of the Shape class, you would get a compile error because a pure virtual function is an abstract class and abstract classes cannot have instances.

Introduction

Before getting into the nitty gritty of copy constructors I think it is best to review a little bit about pointers. Examine the following code snippet:

char str[] = "Hello World";
char *p1 = str;
char *p2 = p1;

In this example two pointers have been created. The pointer p1 is set to the address of the first element of the string str. The pointer p2 is set equal to p1. In this case what is assigned is the address. This means that both p1 and p2 refer to the same memory location that contains str. This type of copy is called a shallow copy. If you were to use cout and print what each pointer refers to they would both print "Hello World". It is important to remember that both pointers refer to the same memory location. What if we simply want a copy of what p1 is pointing at? We could do the following:

char str[]= "Hello World";
char *p1 = str;
char *p2 = new char[strlen(str) + 1];
strcpy(p2, p1);

In this example the pointer p2 gets newly allocated memory and the data that p1 is pointing at is copied into the new memory referred to by p2. This kind of copy is called a deep copy. Now each pointer is pointing at identicaly strings stored in separate memory locations.

Making sure you have a deep copy is the idea behind creating a copy constructor.

Copy Constructor

A copy constructor is a special constructor that takes as its argument a reference to an object of the same class and creates a new object that is a copy. By default, the compiler provides a copy constructor that performs a member-by-member copy from the original object to the one being created. This is called a member wise or shallow copy. Although it may seem to be the desired behavior, in many cases a shallow copy is not satisfactory. To see why let's look at a Person class. We will store the name in a c_string rather than store the name using the string class from the standard C++ library 

#include 

using namespace std;

class Person
{
public:
	Person(const Person &p);
	Person(char *name, int age);
	~Person();
	char *getName()const { return name; }

	//Other Accessor methods 

private:
	char *name;
	int age;
};
Person::Person(char *name, int age)
{
	// Allocate memory for the pointer
	this->name = new char[strlen(name)];
		
	// create the deep copy
	strcpy_s(this->name, strlen(name) + 1, name);
	this->age = age;

}
Person::Person(const Person &p)
{
	// Allocate memory for the pointer
	this->name = new char[strlen(p.name)];

	// create the deep copy
	strcpy_s(this->name, strlen(name) + 1, p.name);
	this->age = p.age;

}
Person::~Person()
{	
	delete[] name;
}
int main()
{
	Person pn("Glenn Stevenson", 21);
	Person programmer(pn);
	cout << programmer.getName() << endl;
	return 0;
}
